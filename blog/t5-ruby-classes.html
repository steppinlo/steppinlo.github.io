Classes Classes Classes


1/24/2015

Well at this point I've been coding consistently for about 5 weeks now. Yet, most of the projects that have been worked on have been pretty short and sweet, so I've never had to think about what happens when things get a bit longer. Sure, I'm familiar with functions/methods and how you can pretty much isolate the method and be able to test it, but what happens when you're working on a complex application and there are loads of methods? How do you keep everything organized? How do you make sure you don't get lost in your own code?? The answer is the usage of #classes.

Classes are great when you want to give methods to your data or have multiple instances of your data. In each class, you can set up your instance or class variables. Class has one copy shared by all different objects in class, and every object has its own copy of an instance variable. In other words, instance variables will have a unique value each time.

So why don't we go ahead and dive right into an example:

class BaseballPlayers def initialize(name, throws, bats, position) @name = name @throws = throws @bats = bats @position = position end def combo "#{@name} throws #{@throws}-handed, bats #{@bats}-handed, and is a #{@position}." end end player1 = BaseballPlayers.new("Madison Bumgarner", "left", "right", "pitcher")

Starting from the top, as you can see we created a class where it would accept information of their name, what hand they bat/throw, and their position. We then introduced the arguments into the code through #initialize, which will run once class is called and sets it up for usage. Here we reassigned the arguments with "@" in front so that it becomes an #instance variable> (if we wanted it to be a class variable, it would be prefixed with "@@"). Next, we've also created a method, and since we already established the arguments in #initialize, we did not need to note it on the same line as #def print. On the bottom we created three instances by creating player1, player2, and player3, with the information in the parenthesis.

Of course nothing will print out as we havent called the instance. But how do we call something within a class? It's pretty simple:

puts player1.combo #Output: Madison Bumgarner throws left-handed, bats right-handed, and is a pitcher.

Very simple! It's actually a lot like arrays and hashes (both of which are classes as well! Makes sense since they're great for organization). As you can see, methods within classes are exactly the same thing with the methods we're familiar with.

Another cool thing with classes, you can have a parent-child relationship! In other words, you can use information from one class in another, or in the right jargon, inheritance. Let's go ahead and create that situation:

class Team < BaseballPlayers def print_all if @name = "Madison Bumgarner" super + " His current team is the San Francisco Giants." end end end

There are a few things we introduced in this portion of the code. First, you can note we created a brand new class called Team. Secondly, we have a < character, which is used when we want a class to be extended (here we can say BaseballPlayers is extended by Team). In essence, our class Team has access to all of BaseballPlayers. As you have probably noted, the method name print_allis also the same as in the BaseballPlayer class. This is because we want to extend the code. By using super, we are using what already occurs in BaseballPlayers.print_all, and then extend it with the rest of the code. So the outcome is below:

player1b = Team.new("Madison Bumgarner", "left", "right", "pitcher") puts player2b.print_all #Outcome: Madison Bumgarner throws left-handed, bats right-handed, and is a pitcher. His current team is the San Francisco Giants.

Also note that we used the if statement, the sole purpose is to show that the variables that we declared in BaseballPlayers is still in play. So, for example, if we had Clayton Kershaw instead, the statement would not be returned/printed. So there you have it, a brief intro to classes!
